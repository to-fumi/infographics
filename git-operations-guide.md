# Git 操作ガイド — はじめての方へ

## このガイドについて

このガイドは **Git をこれから使い始める方** のために書かれています。  
「Git って難しそう…」と感じるかもしれませんが、大丈夫です。日常的に使うコマンドは数個だけですし、慣れればとても頼もしい相棒になります。

ゆっくり、一つずつ進めていきましょう。

---

## 1. Git ってなに？

Git は **ファイルの変更履歴を記録してくれるツール** です。

たとえば、こんな経験はありませんか？

- `企画書_最終版.docx`
- `企画書_最終版_v2.docx`
- `企画書_最終版_v2_修正済み.docx`
- `企画書_本当にこれが最終版.docx`

Git を使うと、こうしたファイルのコピーを何個も作る必要がなくなります。  
**1つのファイルのまま、すべての変更履歴を記録** してくれて、いつでも過去の状態に戻すこともできます。

さらに、チームで開発するときに **「誰が」「いつ」「何を」変えたか** が全部記録されるので、お互いの変更がぶつかって壊れる心配も減ります。

---

## 2. 準備 — Git for Windows のインストール

### ダウンロード

1. [https://gitforwindows.org/](https://gitforwindows.org/) にアクセスする
2. 「Download」ボタンをクリックしてインストーラーをダウンロード
3. ダウンロードしたファイルを実行する

### インストール時の設定

インストーラーでいくつか質問されますが、基本的に **デフォルトのまま「Next」で進めて OK** です。  
迷ったときのために、特に重要な画面だけ補足します。

| 画面 | 推奨選択 | 補足 |
|------|----------|------|
| Default Editor | お好みのエディタを選択 | VS Code を使っている方は「Use Visual Studio Code as Git's default editor」を選ぶ |
| Adjusting your PATH | **Git from the command line and also from 3rd-party software**（デフォルト） | これを選ぶと PowerShell やコマンドプロンプトから `git` コマンドが使える |
| Line ending conversions | **Checkout Windows-style, commit Unix-style line endings**（デフォルト） | Windows と他の OS でファイルを共有するときに改行コードの問題を防いでくれる |

### インストール確認

インストールが終わったら、**PowerShell** または **Git Bash** を開いて、次のコマンドを入力してください。

```bash
git --version
```

`git version 2.xx.x.windows.x` のような表示が出れば成功です。おめでとうございます！

## 3. 最初の設定 — 自分の名前とメールアドレスを登録する

Git を使い始める前に、**「この変更は誰がやったか」** を記録するための設定を行います。  
これは最初に一度だけやればOKです。

```bash
git config --global user.name "あなたの名前"
git config --global user.email "your-email@example.com"
```

> **メールアドレスについて:** GitHub Enterprise（GHE）に登録しているメールアドレスと同じものを使ってください。異なるアドレスだと、GitHub 上でコミットが自分のアカウントと紐づかないことがあります。

設定を確認するには：

```bash
git config --global --list
```

`user.name` と `user.email` が正しく表示されていれば完了です。

## 4. リポジトリを自分のPCにコピーする — `git clone`

GitHub にあるリポジトリを自分の PC にコピーすることを **クローン** と呼びます。

```bash
git clone https://your-ghe-server.com/your-org/your-repo.git
```

> **URL はどこにある？** GitHub のリポジトリページで緑色の「Code」ボタンをクリックすると、URL が表示されます。HTTPS の URL をコピーしてください。

実行すると、リポジトリ名と同じフォルダが作られます。

```bash
cd your-repo
```

これで、あなたの PC にリポジトリの完全なコピーができました。  
ここからは、このフォルダの中で作業していきます。

---

## 5. ブランチの操作 — 安全な作業スペースを作る

### ブランチとは？

ブランチは **「作業用のコピー」** のようなものです。

本のたとえで考えてみましょう。  
共有の原稿（`develop` ブランチ）にいきなり書き込むと、他の人の作業と混ざって大変です。  
そこで、自分専用の下書き用ノート（`feature` ブランチ）を作って、そこで作業します。  
完成したら、下書きの内容を原稿に反映（マージ）する、という流れです。

### 今いるブランチを確認する

```bash
git branch
```

`*` がついているのが、今自分がいるブランチです。

```
  develop
* main
```

この例では `main` ブランチにいることがわかります。

### develop ブランチに移動する

作業を始める前に、まず `develop` ブランチに移動しましょう。

```bash
git checkout develop
```

> **`checkout` ってなに？** 「そのブランチに移動する」という意味のコマンドです。本棚から別のノートを取り出すイメージです。

### 新しいブランチを作って移動する

`develop` ブランチにいる状態で、自分の作業用ブランチを作ります。

```bash
git checkout -b feature/add-login-form
```

`-b` は「**新しく作って、そこに移動する**」というオプションです。  
この1行で「ブランチを作る」と「そこに移動する」の2つを同時にやっています。

> **ブランチ名の付け方** は「GitHub リポジトリ運用ガイド」を参照してください。  
> 簡単に言うと `feature/やること` `fix/直すこと` `chore/雑務` の形式です。

### ブランチを切り替える

すでにあるブランチに移動するときは `-b` なしで：

```bash
git checkout develop
```

## 6. コードを変更して記録する — `add` → `commit`

### 全体の流れ

Git でコードの変更を記録するには、2つのステップがあります。

```
① ファイルを編集する
     ↓
② 変更を「ステージ」に上げる（git add）
     ↓
③ 変更を「コミット」する（git commit）
```

これは荷造りに似ています。

- **`git add`** = 段ボール箱に荷物を入れる（送るものを選ぶ）
- **`git commit`** = 箱にラベルを貼って封をする（記録として確定する）

### 変更されたファイルを確認する

```bash
git status
```

赤い文字で表示されるファイルが「変更されたけど、まだステージに上げていないファイル」です。

```
Changes not staged for commit:
        modified:   src/App.tsx
        modified:   src/styles/main.css

Untracked files:
        src/components/LoginForm.tsx
```

- **modified** = すでに Git で管理されているファイルが変更された
- **Untracked files** = 新しく作ったファイル（まだ Git が知らない）

### ステージに上げる — `git add`

**特定のファイルだけ上げる場合：**

```bash
git add src/App.tsx
git add src/components/LoginForm.tsx
```

**変更されたファイルをすべて上げる場合：**

```bash
git add .
```

> **`.`（ドット）** は「今いるフォルダの中身すべて」という意味です。

もう一度 `git status` を実行すると、ステージに上がったファイルが緑色で表示されます。

```
Changes to be committed:
        modified:   src/App.tsx
        new file:   src/components/LoginForm.tsx
        modified:   src/styles/main.css
```

### コミットする — `git commit`

```bash
git commit -m "feat: ログインフォームを追加"
```

`-m` の後の `"..."` がコミットメッセージです。

> **コミットメッセージの書き方** は「GitHub リポジトリ運用ガイド」を参照してください。  
> 簡単に言うと `feat: 何をしたか` `fix: 何を直したか` の形式です。

### ここまでの流れをおさらい

```bash
# 1. 変更を確認
git status

# 2. ステージに上げる
git add .

# 3. コミットする
git commit -m "feat: ログインフォームを追加"
```

この3つがセットです。何度でも繰り返して大丈夫です。  
1つの機能を作る中で、区切りのいいところで何回かコミットするのがおすすめです。

---

## 7. 変更を GitHub に送る — `git push`

コミットは今のところ **自分の PC の中だけ** に記録されています。  
これを GitHub に送るのが `push` です。

```bash
git push origin feature/add-login-form
```

- **`origin`** = 「GitHub にあるリポジトリ」を指す名前（クローン時に自動で設定される）
- **`feature/add-login-form`** = 送り先のブランチ名

初めてそのブランチを Push するとき、GitHub 側にもブランチが自動で作られます。

### Push したら次は PR

Push が成功したら、GitHub の画面で **Pull Request（PR）** を作成します。  
PR の作り方は次のセクションで説明します。

---

## 8. Pull Request（PR）を作成する

### PR の作り方

1. GitHub でリポジトリのページを開く
2. Push した直後なら、画面の上に **「Compare & pull request」** という黄色いバナーが表示されるのでクリック
3. バナーが出ていない場合は、「Pull requests」タブ → 「New pull request」をクリック

### PR 作成画面で確認すること

| 項目 | 確認内容 |
|------|----------|
| base | マージ先。通常は `develop` |
| compare | 自分の作業ブランチ（例: `feature/add-login-form`） |
| タイトル | コミットメッセージと同じ形式（例: `feat: ログインフォームを追加`） |
| 説明欄 | テンプレートに沿って記入する |

> **base が `main` になっていないか注意！** デフォルトで `main` が選ばれている場合があります。通常の開発では `develop` に向けて PR を出してください。

### PR を出した後の流れ

```
PR 作成
  ↓
CI（自動チェック）が走る ← Lint、テスト、セキュリティスキャン
  ↓
GitHub Copilot がレビューコメントを付ける
  ↓
すべてのチェックが通っている（緑色 ✓）ことを確認
  ↓
「Merge pull request」ボタンをクリック
  ↓
マージ完了！ feature ブランチは自動で削除される
```

### CI が失敗した場合（赤い ✗ が出たとき）

慌てなくて大丈夫です。よくあることです。

1. 失敗しているチェックの「Details」をクリックしてログを確認する
2. エラーの内容を読んで修正する（よくある原因は下記参照）
3. 修正したら再度 `add` → `commit` → `push` する
4. Push すると自動で CI が再実行される

**よくある失敗の原因：**

| チェック | よくある原因 | 対処法 |
|----------|-------------|--------|
| Lint | インデントやフォーマットのずれ | `ruff format .` や `npx prettier --write .` で自動修正 |
| Test | テストが通らない | エラーメッセージを読んで該当のコードを修正 |
| SCA (Trivy) | ライブラリに脆弱性がある | 該当ライブラリのバージョンをアップデート |

## 9. 最新の変更を取り込む — `git pull`

他の人（または自分の別ブランチ）が `develop` に変更をマージした場合、自分の PC にはまだその変更が反映されていません。最新の状態を取り込むには `pull` を使います。

### develop の最新を取り込む

```bash
# develop ブランチに移動
git checkout develop

# 最新の変更を取り込む
git pull origin develop
```

### 作業中のブランチに develop の最新を取り込む

自分の作業が長引いていて、途中で `develop` が更新された場合、最新の `develop` の内容を取り込みたいことがあります。

```bash
# develop の最新を取得
git checkout develop
git pull origin develop

# 自分のブランチに戻る
git checkout feature/add-login-form

# develop の最新を取り込む
git merge develop
```

> **コンフリクト（衝突）が起きたら？** 次のセクションで説明します。

## 10. コンフリクト（衝突）の解消

### コンフリクトとは？

同じファイルの同じ場所を、自分と別の人（または別のブランチ）が同時に変更した場合、Git はどちらの変更を採用すればいいかわかりません。この状態を **コンフリクト（衝突）** と呼びます。

怖い名前ですが、落ち着いて対処すれば大丈夫です。

### コンフリクトが起きたファイルの見え方

コンフリクトが起きると、ファイルの中にこのような印が入ります。

```
<<<<<<< HEAD
<h1>ようこそ</h1>
=======
<h1>こんにちは</h1>
>>>>>>> develop
```

- `<<<<<<< HEAD` から `=======` までが **自分の変更**
- `=======` から `>>>>>>> develop` までが **取り込もうとした側の変更**

### 解消の手順

1. ファイルを開いて、`<<<<<<<`、`=======`、`>>>>>>>` の印を **すべて消す**
2. 最終的に残したい内容だけを残す
3. 保存する

たとえば、「こんにちは」の方を採用する場合：

```
<h1>こんにちは</h1>
```

両方の内容を組み合わせることもできます：

```
<h1>ようこそ — こんにちは</h1>
```

4. 解消したら、通常どおり `add` → `commit` する

```bash
git add .
git commit -m "fix: developとのコンフリクトを解消"
```

> **ヒント:** VS Code を使っている場合、コンフリクトが起きたファイルを開くと「Accept Current Change」「Accept Incoming Change」「Accept Both Changes」というボタンが表示されます。ボタンをクリックするだけで解消できるので便利です。

## 11. 間違えたときの対処法

Git は履歴を記録するツールなので、間違えても元に戻せることが多いです。  
ここでは「よくあるうっかり」とその対処法を紹介します。

### まだ `add` していない変更を元に戻したい

```bash
# 特定のファイルの変更を元に戻す
git checkout -- src/App.tsx

# すべての変更を元に戻す
git checkout -- .
```

> **注意:** この操作は元に戻せません。変更内容が本当に不要か確認してから実行してください。

### `add` したけど、まだ `commit` していない場合

ステージから降ろす（`add` を取り消す）だけなら：

```bash
git reset src/App.tsx
```

ファイルの内容はそのまま残ります。ステージから降りるだけです。

### コミットメッセージを間違えた

直前のコミットメッセージを修正するには：

```bash
git commit --amend -m "feat: 正しいメッセージ"
```

> **注意:** すでに `push` した後にこれをやると、次の Push でエラーになることがあります。Push 前のコミットにだけ使ってください。

### 間違えて `main` にコミットしてしまった

落ち着いて、以下の手順で対処できます。

```bash
# 1. 今のコミットを覚えておく（最新の1コミット分を別ブランチに移す）
git branch temp-branch

# 2. main のコミットを1つ戻す
git reset --hard HEAD~1

# 3. 作業用ブランチに移動する
git checkout temp-branch

# 4. 必要なら正しいブランチ名に変更する
git checkout -b feature/correct-branch-name
```

> もし操作に自信がないときは、無理せずチームメンバーや管理者に相談してください。Git の操作は取り返しがつくことがほとんどです。

## 12. よく使うコマンド早見表

日常的に使うコマンドは実はそれほど多くありません。  
この早見表を手元に置いておけば、ほとんどの作業はカバーできます。

### 基本の流れ（毎日使うもの）

| やりたいこと | コマンド |
|-------------|---------|
| リポジトリをコピーする | `git clone <URL>` |
| 今いるブランチを確認する | `git branch` |
| ブランチを切り替える | `git checkout <ブランチ名>` |
| 新しいブランチを作って移動する | `git checkout -b <ブランチ名>` |
| 変更されたファイルを確認する | `git status` |
| 変更をステージに上げる | `git add .` |
| コミットする | `git commit -m "メッセージ"` |
| GitHub に送る | `git push origin <ブランチ名>` |
| 最新の変更を取り込む | `git pull origin <ブランチ名>` |

### 変更の確認

| やりたいこと | コマンド |
|-------------|---------|
| 変更内容の差分を見る | `git diff` |
| ステージ済みの差分を見る | `git diff --staged` |
| コミット履歴を見る | `git log --oneline` |

### 取り消し・修正

| やりたいこと | コマンド |
|-------------|---------|
| ファイルの変更を元に戻す（add前） | `git checkout -- <ファイル名>` |
| ステージを取り消す（add後、commit前） | `git reset <ファイル名>` |
| コミットメッセージを修正する（push前） | `git commit --amend -m "新しいメッセージ"` |

## 13. 困ったときは

### エラーメッセージが出たら

Git のエラーメッセージは英語ですが、実はかなり親切に書かれています。  
特に `hint:` で始まる行には、「こうすればいいよ」という提案が含まれていることが多いです。

### よくあるエラーと対処法

**`error: failed to push some refs to ...`**

→ GitHub 側に自分の PC にない変更がある状態です。まず `pull` してから `push` してください。

```bash
git pull origin <ブランチ名>
git push origin <ブランチ名>
```

**`error: Your local changes to the following files would be overwritten`**

→ 手元に未コミットの変更があるのに `pull` や `checkout` しようとした場合です。先に `add` → `commit` するか、変更を一時退避してください。

```bash
# 一時退避する場合
git stash
git pull origin develop
git stash pop
```

> **`stash` ってなに？** 「変更を一時的にしまっておく」コマンドです。引き出しに一旦しまって、後で取り出すイメージです。`stash pop` で取り出せます。

**`CONFLICT (content): Merge conflict in ...`**

→ コンフリクトが発生しています。セクション10「コンフリクトの解消」を参照してください。

### それでも解決しないとき

- **エラーメッセージをそのままコピーして検索** すると、同じ問題に遭遇した人の解決策が見つかることが多いです
- **チームメンバーや管理者に相談** してください。Git のトラブルはよくあることなので、恥ずかしいことではありません
- **最終手段:** フォルダを削除してもう一度 `git clone` からやり直すこともできます。Push 済みの変更は GitHub に残っているので安心してください

---

## おわりに

最初は覚えることが多く感じるかもしれませんが、日常的に使うのは本当に数個のコマンドだけです。

```
git checkout -b    → ブランチを作る
git add .          → 変更をステージに上げる
git commit -m      → コミットする
git push origin    → GitHub に送る
git pull origin    → 最新を取り込む
```

この5つさえ覚えれば、日々の開発は回せます。  
残りは必要になったときにこのガイドを見返せば大丈夫です。

使っていくうちに自然と慣れていきますので、安心して取り組んでください。
